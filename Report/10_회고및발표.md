## 요약

6시간 실습에서 README 순서대로 **분석 → 1차 리팩토링 → 단위 테스트 → 기능 개선 → 회고**를 진행했다. 처음엔 `calculateBmi` 한 함수에 모든 일이 들어 있고 테스트도 `FAIL()`뿐이었는데, 지금은 `ctest` **31/31 Green**까지 왔다 (Report 09). AI는 코드 작성보다 **범위를 좁혀 달라고 부탁할 때** 가장 도움이 됐고, BMI=25 같은 **경계값은 내가 README와 대조하지 않으면 놓쳤다** (DEF-001, Report 07·08).

## 2. 코드 품질 Before & After

### 2.1 Before — 내 말로 다섯 줄

1. **`calculateBmi` 한 함수에 모든 일**이 들어 있어서(약 100줄, Report 03) 어디서 버그가 났는지 찾기 힘들었다.
2. **18.5, 23, 25, 100~400** 같은 숫자가 코드 곳곳에 박혀 있어(매직 넘버) 의미를 외우지 않으면 읽기 어려웠다.
3. 연령대마다 **비슷한 if-else가 여러 번** 반복됐다 (Report 01·03).
4. README는 **BMI ≥ 25 비만**인데, 코드는 `> 25`라서 25.0에서 어긋날 여지가 있었다 (Report 01 §2.1).
5. 테스트 파일은 **`FAIL()` 스텁만** 있어서, 고칠 때마다 “망가졌는지” 확인할 방법이 없었다.

### 2.2 After — 내가 본 변화

- **Facade + 파이프라인:** `calculateBmi`는 파일 로드 후 `runBmiPipeline()`만 호출한다 (Report 09).
- **상수·타입:** `SHealthConstants`, `BmiCategoryCode`, `AgeBandDistribution` 등으로 숫자 의미가 이름에 붙었다 (Report 04).
- **public API 5종:** 기존 `calculateBmi`·`getBmiRatio`에 더해 F-09~12 API가 추가됐다 (Report 09).
- **분류 버그 수정:** `classifyBmi`에서 `bmi >= kBmiOverweightMax` — DEF-001 Fixed (Report 08).
- **검증:** **`ctest` 31/31 Green** (Report 09).

### 2.3 비교 표

| 항목 | Before | After | 내가 느낀 변화 |
|------|--------|-------|----------------|
| 구조 | God Method 1개 | `loadFromStream` + `runBmiPipeline` | “어디를 고칠지” 범위가 줄었다 |
| 상수 | 매직 넘버 | `SHealthConstants` | TC 작성 시 README 표와 맞추기 쉬워졌다 |
| 분류 | `> 25` (25.0 누락) | `>= 25` | TC_16이 **알려줘서** 고쳤다 |
| height=0 | BMI `inf` 쪽 | `imputeMissingHeightsByAgeBand` (F-10) | TC_34로 동작이 고정됐다 (DEF-003) |
| 테스트 | `FAIL()` | 31 `TEST_F`, 31/31 Green | 리팩토링할 때 **덜 무서웠다** |
| 기능 | 연령대 비율 위주 | F-09~12 + main 3종 출력 | README 4단계 요구를 코드로 확인했다 |

## 3. AI 활용 — 도움 vs 한계

### 3.1 도움이 된 순간 (3)

1. **한 턴에 한 가지만** — 2단계에서 “네이밍만”, “상수화만”처럼 범위를 적어 주면, `./SHealthBMI` 6연령대 출력이 안 바뀌는지 확인하기 쉬웠다 (Prompting/04, Report 04). 나는 “한 번에 다 고쳐줘”보다 **순서대로 부탁**하는 쪽이 안전했다.

2. **TC_16 Red → 결함 정리 → Green** — 테스트 25/26일 때 AI에게 실패 로그와 `@docs/defect_list.md`를 붙여 **DEF-001**로 원인을 좁혔고, `>=`와 픽스처 weight 72.25만 고쳐 **26/26**이 됐다 (Report 06·07·08). “왜 Red인지”를 문서로 남기는 과정이 특히 도움이 됐다.

3. **@문서로 기준 맞추기** — `@docs/test_plan.md`, `@docs/requirements_analysis.md`, Report 번호를 프롬프트에 넣으면, AI 답과 내가 읽은 SSOT가 어긋날 때 **바로 잡을 수 있었다** (Report 05·07).

### 3.2 한계 · 내가 검증 안 하면 틀렸던 것 (3)

1. **BMI=25 경계** — README는 `≥25` 비만인데, 초기 코드는 `bmi > 25`였다. AI가 “기존 동작 유지” 쪽으로 말하면, **README 표를 내가 먼저 읽지 않으면** 버그를 그대로 둘 뻔했다 (DEF-001, test_plan TC_16).

2. **한 턴 과다 변경** — God Method를 `vector`로 통째로 바꾸거나 여러 파일을 동시에 손대면, `.cursorrules`의 **리팩토링 → 테스트 → 기능** 순서와 부딪혔다. 나중엔 브랜치(`refactoring` / `tc` / `feature`)와 **“금지: …”** 문장을 프롬프트에 넣었다 (Report 04·09).

3. **스냅샷 TC 오해** — TC_07(DEF-002)은 “지금의 이상 동작을 기록”하는 쪽이라, 무조건 Green만 맞추면 **요구 확정 없이 버그를 고친 척** 할 수 있다 (defect_list). AI에게 “TC_07 Green으로 만들어줘”만 하면 위험했다.

### 3.3 다음에 AI에게 부탁할 때 체크리스트 (5)

1. **역할·단계**를 한 줄로: “2단계 리팩토링만”, “F-09만” (Report 04·09).
2. **@로 SSOT 첨부:** `test_plan`, `defect_list`, 해당 Report.
3. **금지 사항**을 bullet로: `BMI>=25 수정 금지`, `DEF-001 재수정 금지` 등.
4. **완료 조건**을 숫자로: `ctest 26/26 유지`, `./SHealthBMI` 기준선 동일.
5. **출력 형식** 요청: 변경 표 + diff + 확인 체크리스트 — 나는 diff를 **직접 훑어본 뒤** 빌드한다.

> **한 줄 교훈:** AI는 **코드 공장**이 아니라 **범위를 지키게 해 주는 조수**에 가깝고, 맞는지는 **ctest와 README**가 최종 판단한다.

## 4. TC 추가가 개선에 미친 영향 + TC 작성 팁

### 4.1 스토리 — 테스트가 이끈 개선

**1) TC_16 → DEF-001 (분류 경계)**

- 26개 TC 중 **TC_16만 Red**였다 (Report 06). 기대는 비만 100%인데 실제는 0% — 로그를 보고 `classifyBmi`의 `>` vs README `≥`를 찾았다 (Report 07).
- 수정은 작게: `bmi >= kBmiOverweightMax` + `tc16` CSV weight **72.25** (Report 08) → **26/26 Green**.
- 이어서 **TC_18**으로 4분류 합 100% 회귀를 확인했다 (DEF-004, defect_list).

**2) TC_05 · TC_34 → DEF-003 (height=0)**

- height=0이면 BMI가 `inf` 쪽으로 가는 문제를 TC로 본 뒤, 4단계 **F-10** `imputeMissingHeightsByAgeBand`를 넣었다 (Report 09, DEF-003 Fixed).
- **TC_34**가 보정 후 동작을 고정해 줘서, 기능 추가 후에도 깨지지 않게 됐다.

**3) TC_33~37 → F-09~12 (신규 API)**

- 연령대 분포(F-09), 정상 BMI ID 목록(F-11), 전체 비율(F-12)을 API로 올릴 때마다 TC를 붙였다 (Report 09).
- 최종 **31/31 Green** — 3단계 26건에서 5건이 늘었고, 기존 TC가 **회귀 방어막**이 됐다.

**테스트 없었으면 ___를 몰랐을 것**

> 테스트 없었으면 **BMI=25.0이 비만이 아니라 과체중으로 집계된다**는 README와 코드의 어긋남을, `./SHealthBMI` 출력만으로는 끝까지 몰랐을 것이다 (DEF-001).

### 4.2 비기너용 TC 작성 팁 (5)

1. **Given-When-Then 주석**을 `TEST_F` 맨 위에 쓴다 — 나중에 실패 로그를 읽을 때 “무엇을 검증하려 했는지”가 바로 보인다 (Report 05, `.cursorrules`).

2. **경계값은 README 표와 쌍으로** — 18.5 / 23 / 25는 TC_11~17과 `tcNN_*.csv`에 기대 weight·BMI를 적어 둔다 (test_plan §6). TC_16이 그 대표 사례다.

3. **픽스처는 `test/fixtures/`** — `fixturePath("tc16_bmi_25.csv")`, 빌드 시 `SHEALTH_TEST_FIXTURE_DIR`로 경로를 맞춘다 (Report 06).

4. **한 TC = 한 실패 원인** — TC_16만 Red로 두니 Green 턴에서 `>=` 한 줄짜리 수정으로 끝났다 (Report 08). 여러 버그를 한 TC에 넣지 않는 게 좋다.

5. **Red/Green 의도를 이름·defect_list에** — TC_07은 Snapshot(DEF-002), TC_34는 F-10 이후 **의도된 동작 변경**이다 (defect_list). “Green이 목표”인지 “기록이 목표”인지 구분한다.

> **한 줄 교훈:** TC는 숙제가 아니라 **README와 코드가 싸울 때 심판**이고, Red 한 건이 버그 수정 순서를 정해 줬다.

## 5. 클린코드·리팩토링 — 장점 vs 어려운 점

### 5.1 장점 (3)

1. **기준선 출력(`./SHealthBMI`)** — 2단계마다 6연령대 숫자를 비교해, “리팩토링인지 버그인지”를 나눌 수 있었다 (Report 04). 비기너에게는 **자동 테스트 전에 손으로 확인하는 안전벨트** 같았다.

2. **작은 추출이 쌓임** — 처음부터 클래스를 쪼개지 않고 `classifyBmi` 같은 **private 함수**부터 만들었더니, 4단계 SRP(`loadFromStream`, `runBmiPipeline`)로 자연스럽게 이어졌다 (Report 04 → 09). 한 번에 완벽한 설계를 요구하지 않아도 됐다.

3. **테이블·enum으로 중복 감소** — 24개 멤버 if-else가 `AgeBandRatios[6]` 등으로 줄어, F-09~12 API를 붙일 때 **같은 집계 결과를 다른 형태로 꺼내기**만 하면 됐다 (Report 04 §2.4, 09). “복붙 지옥”이 줄어든 게 체감상 가장 컸다.

### 5.2 어려운 점 (3)

1. **God Method 잔재** — `ids[10000]` 같은 고정 배열은 그대로라, “객체지향적으로 잘 나눴다”고 말하기는 아직 부끄럽다 (Report 03). 어디까지 손대도 되는지 **`.cursorrules` 범위**를 자주 확인했다.

2. **버그를 고치고 싶은데 순서가 막힘** — BMI≥25는 리팩토링 중에 고치고 싶었지만, 규칙상 **테스트를 만든 뒤**(TC_16 Red) 수정하는 흐름이 필요했다 (Report 04 §1.2, 06). 답답했지만, 나중에 “왜 Red였는지” 설명할 수 있게 됐다.

3. **API가 이중으로 느껴짐** — `getBmiRatio(age, 100~400)` 레거시와 `getAgeBandDistribution` struct가 공존한다 (Report 09). 동작은 맞지만, **어떤 API를 써야 하는지**는 아직 헷갈린다.

### 5.3 `.cursorrules` 순서를 체감한 이유

규칙은 **리팩토링 → 단위 테스트 → 버그 수정 → 기능 개선**이다.

- 2단계에서 BMI≥25를 고치지 않은 건 처음엔 이해가 안 됐다. 그런데 테스트 없이 고쳤으면 **TC_16이 증거가 되지 못했을 것**이다.
- 3단계에서 Red를 만든 뒤 8단계 Report로 **DEF-001**을 고정하고, 4단계에서 기능을 올릴 때 **31/31**로 회귀를 잡았다.
- 비기너 입장에서는 “느리다”기보다 **증거가 쌓이는 순서**에 가깝다고 느꼈다.

> **한 줄 교훈:** 클린코드는 하루 만에 완성하는 게 아니라, **읽기 쉬운 조각 + 테스트**를 쌓아 가는 과정이었다.
