## 2. 도메인·아키텍처 요약

### 2.1 비즈니스 규칙

- **입력:** `shealth.dat` — `id, age, weight(kg), height(cm)`
- **BMI:** `weight / (height_m)²`, `height_m = height_cm / 100`
- **분류**
  - 저체중: BMI ≤ 18.5
  - 정상: 18.5 < BMI < 23
  - 과체중: 23 ≤ BMI < 25
  - 비만: BMI ≥ 25 (README 기준; 코드 일부는 `> 25`만 비만 처리 — 테스트 작성 시 경계값 명시 필요)
- **보정:** 체중 `0` → 동일 **10년 연령대** `[a, a+10)` 내 유효 체중 평균으로 대체

### 2.2 모듈 구조

```
SHealthBMI.cpp     → main, 실행 진입
SHealth.cpp/.h     → CSV 로드, 보정, BMI·연령대별 비율, getBmiRatio
SHealthBMITest.cpp → Google Test (현재 실패 스텁)
shealth_lib        → CMake 정적 라이브러리
```

## 3. 코드 품질 Before & After

### 3.1 Before (현재 `SHealth` 레거시)

**강점**

- 단일 클래스에 end-to-end 파이프라인(로드 → 보정 → BMI → 통계)이 한눈에 보임
- `split`, 파일 읽기 등 최소 동작은 구현됨

**코드 스멜 (분석 결과)**

| 스멜 | 위치·예시 | 영향 |
|------|-----------|------|
| **Long Method** | `calculateBmi` 한 함수에 로드·보정·계산·집계 | 테스트·재사용 어려움 |
| **Duplicate Code** | 연령대 20~70별 `underweight20`…`obesity70` 및 동일 if-else 블록 | 유지보수 비용, 버그 전파 |
| **Magic Number** | `18.5`, `23`, `25`, `100`/`200`/`300`/`400`, `10000` | 의도 불명확 |
| **Large Class / Data Clump** | 고정 배열 4종 + 통계 멤버 24개 | SRP 위반 |
| **Primitive Obsession** | `getBmiRatio(ageClass, type)`에 `type` 100~400 | API 가독성 저하 |
| **경계 불일치 가능** | `bmis[i] > 25` vs README `25 이상` | 25.0 정확히 경계 시 불일치 위험 |
| **예외·안전** | `ageCount == 0` 시 0으로 나눔, `stoi`/`stod` 예외 미처리 | 런타임 리스크 |
| **테스트 부재** | `TEST(..., FailedTest) { FAIL(); }` | 리팩토링 안전망 없음 |

### 3.2 After (목표 상태)

| 영역 | 목표 |
|------|------|
| 구조 | `loadRecords`, `imputeWeightByAgeBand`, `computeBmi`, `classifyBmi`, `aggregateByAgeBand` 등 책임 분리 |
| 상수 | `kUnderweightMax`, `kNormalMax`, `kOverweightMax`, `enum class BmiCategory` |
| 데이터 | `std::vector<Record>` 또는 연령대×카테고리 2차 구조로 중복 제거 |
| 테스트 | `TEST_F` + Given-When-Then, `ctest` Green 유지하며 리팩토링 |
| AI 작업 | 루트 `.cursorrules`로 단계·규칙 고정 (리팩토링 → UT → 기능) |

### 3.3 정량 비교 (참고)

## 4. 생성형 AI 활용

### 4.1 수행 내용

- **`.cursorrules` 작성:** C++17 + CMake + GTest + gcov/lcov, Green 게이트, Given-When-Then/`TEST_F`, 작업 순서(리팩토링 → UT → 기능)를 프로젝트 루트에 문서화하여 Cursor가 매 세션 동일 규칙을 따르도록 함.

### 4.2 도움이 된 점

- README 6시간 커리큘럼을 **에이전트 행동 규칙**으로 변환 → 단계 혼합(기능 추가를 테스트 전에 하는 등) 방지
- BMI 경계·보정 규칙을 절대 규칙에 넣어 **도메인 드리프트** 완화
- TDD 프롬프트 템플릿(`TDD 기반 … 프롬프트.txt`)과 연계해 2·3단계 질의 일관성 확보

### 4.3 한계·주의

- AI는 **현재 동작을 보존**한다고 가정하나, 레거시 버그(경계값, 0 나눗셈)는 그대로 옮길 수 있음 → **실패 테스트로 기대 동작을 먼저 고정**해야 함
- 대규모 한 번에 리팩토링 제안을 막기 위해 `.cursorrules`에 “한 번에 하나” 명시 필요 (이미 반영)
- Windows/Linux 빌드·lcov 환경 차이는 AI가 추측할 수 있음 → `cmake`/`ctest` 결과는 로컬에서 확인

### 4.4 권장 프롬프트 예 (다음 단계)

1. *「`.cursorrules` 1단계만. `calculateBmi`에서 매직 넘버만 상수화하고 동작은 바꾸지 마. ctest는 아직 Red이니 테스트 파일은 건드리지 마.」*
2. *「`TEST_F(SHealthFixture, …)`로 BMI 계산 Given-When-Then 테스트만 추가해. 실패하는 테스트부터.」*
3. *「Green 상태 확인 후 `getBmiRatio` if-else 체인을 연령대 루프로 리팩토링.」*

## 5. 단위 테스트 전략 및 작성 팁

### 5.1 우선순위 (`.cursorrules` / README 정합)

1. BMI 단일 레코드 계산 (`height_cm`, `weight_kg`)
2. `weight == 0` → 동일 연령대 평균 보정
3. 분류 경계: 18.5, 23, 25 (포함/미포함 각각)
4. 예외: 빈 파일, 헤더만, 잘못된 CSV, 해당 연령대 유효 체중 없음

### 5.2 Given-When-Then + `TEST_F` 예시 골격

```cpp
class SHealthFixture : public ::testing::Test {
protected:
    SHealth health;
};

TEST_F(SHealthFixture, Bmi_IsWeightOverHeightSquared) {
    // Given: height 170cm, weight 70kg (인메모리 또는 최소 CSV 픽스처)
    // When:  BMI 계산 API 호출 (리팩토링 후 노출된 순수 함수)
    // Then:  EXPECT_NEAR(24.22, bmi, 0.01);
}
```

### 5.3 TC가 리팩토링에 미치는 영향 (예상)

| 효과 | 설명 |
|------|------|
| 안전망 | 연령대 루프 통합 시 회귀 즉시 감지 |
| 명세화 | README와 코드의 `25` 경계 불일치를 테스트로 합의 |
| 설계 압력 | `calculateBmi` 분해를 유도 (테스트 가능한 public/ package 함수) |

### 5.4 작성 팁

- **한 테스트 = 한 이유로 실패**; 이름은 `Method_Scenario_Expected` 형식
- 파일 I/O는 **임시 CSV** 또는 픽스처 디렉터리 `Report/fixtures/` (추가 시)로 격리
- 부동소수는 `EXPECT_NEAR` + 허용 오차 명시
- 2단계 전에는 `FAIL()` 스텁 제거 또는 `@Disabled` 대신 **삭제 후 실제 `TEST_F`로 교체**

## 6. 클린코드·리팩토링 회고 (예비)

### 6.1 기대 장점

- 연령대·BMI 카테고리 중복 제거 시 **버그 수정 한 곳**으로 수렴
- 상수화로 **요구사항 변경**(WHO 기준 등) 대응 용이
- SRP 적용 시 4단계 기능(목록 조회, 전체 비율) 추가가 수월

### 6.2 예상 어려움

- **동작 보존 vs 버그 수정:** README와 구현이 다를 때 테스트를 “현재 동작”에 맞출지 “올바른 동작”에 맞출지 결정 필요
- **private 배열:** 테스트를 위해 순수 함수 추출 또는 `friend` / 테스트 전용 헤더 — 최소 침습 선택
- **Green 게이트:** 초기에는 Red → 첫 Green까지 시간 투자 필요
